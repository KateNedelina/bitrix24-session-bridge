# Bitrix24 Session Bridge

Сессионный мост для Bitrix24 CRM, который помогает агенту войти в портал через обычную веб-форму, удерживать браузероподобную сессию и собирать полный рабочий контекст по компании: карточки, сделки, timeline, связанные сущности, документы, снимки страниц, метаданные и человекочитаемый `context.md`.

Этот skill нужен там, где REST webhook или OAuth ещё не настроены, недоступны текущему пользователю или требуют проверки через живой интерфейс Bitrix24.

## Когда использовать

Используйте `bitrix24-session-bridge`, если нужно получить CRM-контекст по компании из self-hosted или on-prem Bitrix24, а нормальный REST-доступ пока отсутствует.

Типичные задачи:

- собрать всё доступное в CRM по одной компании;
- проверить, открываются ли разделы интеграций, webhook и локальные приложения;
- прочитать карточку компании, сделки или контакта по прямой ссылке;
- найти сделки и компании через CRM-гриды;
- сохранить доказательную базу на диск для дальнейшего анализа, memo, презентации или outreach.

Предпочтительный порядок такой: сначала webhook или OAuth, затем этот сессионный мост как временный, проверочный или аварийный путь.

## Что собирает skill

В рабочем сценарии skill формирует dossier по компании и сохраняет:

- карточки компаний, найденные по названию или точному `company_id`;
- сделки из грида, карточки компании, timeline и redirect-ссылок на `/crm/deal/details/<id>/`;
- вкладки карточки компании и технические loader URL;
- ленивые вкладки сделок, доступные через `serviceUrl`;
- связанные сущности из timeline: сделки, договоры, ДС, заявки и другие smart-process карточки;
- HTML/TXT-снимки карточек и вкладок;
- коммуникации связанных контактов;
- документы из CRM user-field файлов, прямых ссылок и Bitrix Disk папок;
- машинные метаданные и человекочитаемый `context.md`.

## Быстрый старт

1. Скопируйте пример окружения:

```bash
cp .env.example .env
```

2. Заполните `.env` локальными доступами к Bitrix24:

```dotenv
B24_BASE_URL=https://your-bitrix24.example.com
B24_LOGIN=your.login@example.com
B24_PASSWORD=your-password
```

3. Проверьте вход и базовые разделы:

```bash
python3 scripts/bitrix24_session_client.py probe
```

4. Соберите dossier по компании:

```bash
python3 scripts/bitrix24_session_client.py build-company-dossier \
  'Название компании' \
  --mode full
```

5. Проверьте результат:

```bash
cat bitrix24_company_contexts/<company-slug>/metadata/run_report.json
open bitrix24_company_contexts/<company-slug>/context.md
```

## Переменные окружения

Скрипт автоматически читает файл `.env` из корня skill.

| Переменная | Обязательна | Назначение |
|------------|-------------|------------|
| `B24_BASE_URL` | да | Базовый URL портала Bitrix24 без завершающего `/`. |
| `B24_LOGIN` | да | Логин пользователя Bitrix24. |
| `B24_PASSWORD` | да | Пароль пользователя Bitrix24. |

Файл `.env` должен оставаться только локальным. Для публичного репозитория используйте `.env.example`.

## Основные команды

Проверить логин и доступность типовых разделов:

```bash
python3 scripts/bitrix24_session_client.py probe
```

Получить авторизованную страницу как текст:

```bash
python3 scripts/bitrix24_session_client.py fetch \
  /crm/deal/details/14325/ \
  --format text
```

Извлечь ссылки со страницы:

```bash
python3 scripts/bitrix24_session_client.py fetch \
  /crm/company/list/ \
  --format links
```

Прочитать сделки через облегчённый канал Bitrix-грида:

```bash
python3 scripts/bitrix24_session_client.py list-deals
```

Отфильтровать сделки по подстроке в клиенте:

```bash
python3 scripts/bitrix24_session_client.py list-deals \
  --client-contains 'Название клиента' \
  --max-pages 25
```

Найти карточки компаний:

```bash
python3 scripts/bitrix24_session_client.py list-companies \
  --name-contains 'Название клиента'
```

Собрать контекст по названию компании:

```bash
python3 scripts/bitrix24_session_client.py build-company-dossier \
  'Название клиента' \
  --mode full
```

Собрать контекст по известному `company_id`:

```bash
python3 scripts/bitrix24_session_client.py build-company-dossier \
  --company-id 12345 \
  'Название клиента' \
  --mode full
```

## Режимы сбора

| Режим | Когда использовать | Что делает |
|-------|--------------------|------------|
| `quick` | Быстрая проверка, первичная разведка, fallback после тяжёлого запуска. | Собирает базовые карточки, найденные сделки, основные снимки и документы без расширенного раскрытия вкладок. |
| `full` | Режим по умолчанию для рабочих CRM-досье. | Собирает карточки, сделки, ленивые вкладки, связанные сущности, документы и метаданные. |
| `deep` | Осторожный расширенный сбор, когда нужны дополнительные ленивые вкладки и связанные карточки. | Делает больше запросов к порталу и может работать заметно дольше. |

## Структура результата

По умолчанию результат сохраняется в текущую директорию:

```text
bitrix24_company_contexts/
└── <company-slug>/
    ├── context.md
    ├── raw/
    ├── documents/
    └── metadata/
        ├── communications.tsv
        ├── company_details.json
        ├── deal_details.json
        ├── deal_matches.json
        ├── document_entrypoints.json
        ├── documents.json
        ├── entity_links.json
        ├── lazy_tabs.json
        ├── related_entities.json
        ├── run_report.json
        ├── tabs.json
        └── timeline_highlights.json
```

Главный файл для человека — `context.md`. Технические URL, loader-пути, массовые выгрузки контактов, HTML-снимки и подробные реестры должны оставаться в `raw/` и `metadata/`.

После каждого запуска обязательно проверяйте:

- `metadata/run_report.json` — итоговый статус, режим, ошибки, предупреждения, количество страниц, вкладок и документов;
- `context.md` — основной читабельный контекст;
- `metadata/lazy_tabs.json` — какие ленивые вкладки удалось раскрыть.

## Контракт для агентов

Агент, использующий этот skill, должен работать только на русском языке, кроме технических идентификаторов, URL, имён полей, путей, команд и исходных фрагментов интерфейса Bitrix24.

Обязательный порядок для CRM-dossier:

1. Запустить `probe` и убедиться, что сессия открывает портал, а не форму авторизации.
2. Если пользователь дал точный `company_id`, использовать его как основной якорь.
3. Если известна только компания, сначала найти карточки через `list-companies`.
4. Собрать dossier через `build-company-dossier --mode full`.
5. Проверить `metadata/run_report.json`, `context.md` и `metadata/lazy_tabs.json`.
6. Для сделок использовать card-only attribution: не смешивать найденные глобальным поиском сделки с карточно подтверждёнными без явного указания источника.
7. Не копировать в чат сырые HTML, массовые контакты, loader URL и внутренние endpoint-детали, если пользователь не просит именно техническую диагностику.
8. Не сохранять в skill одноразовые бизнес-выводы, имена клиентов или временные аналитические находки.

Если `full` долго не даёт видимого прогресса, проверьте `metadata/run_report.json`, рост `context.md` и состояние процесса. При необходимости используйте `quick` как контролируемый fallback.

## Безопасность

- Никогда не коммитьте `.env`, cookies, реальные CRM-выгрузки и скачанные документы.
- Никогда не печатайте пароль в терминале, README, issue, PR или финальном ответе агента.
- Публичный репозиторий должен содержать только код, инструкции и безопасные примеры.
- CRM-документы и `bitrix24_company_contexts/` относятся к рабочим данным и должны храниться отдельно от публичной кодовой базы.
- DOM-селекторы, `bxajaxid`, `gridId`, `serviceUrl` и внутренние Bitrix URL считайте нестабильными деталями реализации.

## Ограничения

- Skill не заменяет полноценную Bitrix24 REST-интеграцию.
- Веб-сессия и HTML-парсинг хрупче webhook/OAuth.
- Портальные интерфейсы, гриды, вкладки и AJAX endpoint могут меняться без предупреждения.
- У пользователя Bitrix24 должны быть права на соответствующие карточки, документы и разделы.
- Self-hosted и on-prem инсталляции Bitrix24 могут отличаться настройками, кастомными полями и smart-process сущностями.

## Диагностика

Если команда возвращает форму авторизации, проверьте `B24_BASE_URL`, `B24_LOGIN`, `B24_PASSWORD` и доступность портала из текущей сети.

Если карточка выглядит пустой, попробуйте прямой detail URL и формат `html`, затем ищите `serviceUrl`, `gridId`, `bxajaxid`, `sessid` и компонентные AJAX-пути.

Если список сделок тяжёлый, предпочитайте grid fragment path вместо полного reload:

```text
/crm/deal/list/?by=&order=&page=N&bxajaxid=<value>
```

Если документы не скачались, проверьте `metadata/document_entrypoints.json`, `metadata/documents.json` и предупреждения в `metadata/run_report.json`.
